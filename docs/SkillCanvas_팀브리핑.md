## 1. 용어 (프론트가 알아야 할 최소한)

| 용어 | 쉽게 |
|------|------|
| **스킬(Skill)** | Claude에게 시키는 "나만의 기능" (프롬프트 덩어리). 노드 1개. |
| **MCP** | Claude가 외부 도구(Gmail·택배·노션)에 손을 뻗는 **연결 어댑터**. 노드 1개. |
| **에이전트(Agent)** ⭐ | 스킬+도구로 **판단·생성하는 Claude 두뇌 노드** (예: "긴급도 판단"). |
| **하네스(Harness)** ⭐ | 워크플로우가 **믿을 수 있게 돌게** 하는 제어 장치 (중복체크·검증·**승인**). |
| **트리거(Trigger)** | 워크플로우를 언제 시작할지 (예: 매일 9시). |
| **규칙(Rule)** | 제약 (예: "요약 200자"). |
| **워크플로우** | 위 노드들을 순서대로 엮은 **자동화 흐름** (= 캔버스에 그리는 그래프). |
| **로컬 실행기** | 사용자 컴퓨터에서 도는 작은 프로그램. **파일 읽고 쓰고, Claude·도구를 실제로 실행.** |
| **카탈로그** | 우리가 지원하는 MCP/도구 목록(레지스트리). 여기 있는 것만 추천·연결. |
| **`.claude`** | 사용자 컴퓨터 안, 스킬·MCP 설정 파일들이 있는 폴더. |

## 2. 아키텍처 — 프론트는 "두 군데"랑 통신한다 (중요!)

프론트(브라우저)는 **서버 하나가 아니라 두 곳**과 통신해요:

```
┌────────────────────────────────────────────────────────┐
│  🖥️ 프론트 (React + React Flow, Vercel 배포)             │
│     화면·노드캔버스·팝업                                  │
└───────┬──────────────────────────────────┬─────────────┘
        │ ① 인터넷(REST)                    │ ② localhost(REST)
        ▼                                   ▼
┌──────────────────────┐        ┌──────────────────────────────┐
│ 호스팅 백엔드(FastAPI) │        │ 로컬 실행기(FastAPI, 사용자 PC) │
│ · 갤러리 (공유)        │        │ · .claude 파일 읽고 쓰기       │
│ · MCP 카탈로그         │        │ · 워크플로우 실제 실행         │
│ · 자연어→그래프 조립   │        │ · 하네스(중복·검증·승인)       │
│ · 로그인(Clerk) 검증   │        │ · Claude·MCP 호출 · 스케줄     │
│ → DB: RDS(Postgres)   │        │ → 로컬 SQLite(실행상태·이력)   │
└──────────────────────┘        └──────────────────────────────┘
```

**프론트가 어디에 뭘 요청하나:**
| 기능 | 어디로 |
|------|--------|
| 자연어 → 워크플로우 조립(AI 추천) | ① 호스팅 백엔드 |
| 갤러리(올리기·가져오기·목록) | ① 호스팅 백엔드 |
| MCP 카탈로그 목록 | ① 호스팅 백엔드 |
| 로그인 | Clerk |
| **내 `.claude` 파일 읽기/저장(동기화)** | ② 로컬 실행기 |
| **워크플로우 실행 / 승인 / 상태** | ② 로컬 실행기 |

> ⚠️ 핵심: **파일·실행은 서버가 아니라 "사용자 컴퓨터의 로컬 실행기"가 함.** 웹이 직접 파일을 못 만지기 때문(브라우저 보안). 로컬 실행기는 사용자가 다운로드해서 켜둠. 개인 데이터(메일·처리이력)는 **로컬에만** 있고 우리 서버로 안 감.

## 3. 백엔드 흐름 — 두 단계 (전체가 어떻게 도는지)

### 【A】 조립 단계 — 사용자가 말로 만들 때
```
① 사용자가 자연어 입력 ("나 CS 담당인데 컴플레인 오면...")
② 프론트 → 호스팅 백엔드로 전송
③ 백엔드가 Claude 호출 → "카탈로그 안 도구로 워크플로우 그래프 만들어"
④ 백엔드가 검증(카탈로그에 있는 도구인지, 그래프 유효한지)
⑤ 프론트가 그래프(JSON)를 받아 React Flow로 노드 렌더
⑥ 사용자가 노드 보고 드래그·편집
⑦ "저장" → 로컬 실행기가 .claude에 씀 (양방향 동기화)
```
→ **프론트 할 일**: 자연어 입력 UI → 받은 그래프 JSON을 노드로 그리기 → 편집 → 저장 요청.

### 【B】 실행 단계 — 워크플로우가 실제로 도는 순간 (데모 하이라이트)
```
로컬 실행기가 그래프를 노드 순서대로 실행:
① 트리거(스케줄) 시작
② MCP: Gmail에서 컴플레인 읽기
③ 에이전트: 긴급도 판단 ("화남 95%")
④ 하네스-중복체크: 로컬 SQLite 조회 → 어제 처리함? 스킵/진행
⑤ MCP: 택배 조회 ("대전 HUB 체류중")
⑥ 하네스-검증: 진짜 배송지연 맞나 → 통과해야 진행
⑦ 에이전트: 사과+쿠폰 문구 생성
⑧ 하네스-🚨승인게이트: 실행 멈춤 → 사용자에게 팝업 → [승인] 기다림
⑨ (승인 후) MCP: Gmail 사과메일 발송
⑩ 하네스-기록: 로컬 SQLite에 "처리완료" 저장
```
→ **프론트 할 일 (제일 중요)**: 실행되는 동안 **노드들의 상태가 순서대로 바뀌는 걸 보여주고**(대기→실행중→완료), **⑧ 승인게이트에서 멈추면 승인 팝업**을 띄우고, 사용자가 [승인]하면 계속 진행되는 걸 보여주기.

## 4. 노드 타입 (프론트가 캔버스에 그릴 것)

현재 목업엔 스킬/MCP/규칙/트리거만 있음. **에이전트 + 하네스 노드 추가 필요.**

| 노드 | 색(제안) | 역할 | 상태 |
|------|---------|------|------|
| **트리거 (Trigger)** | 초록 | 언제 시작 (스케줄 등) | 유지 |
| **스킬 (Skill)** | 파랑 | 나만의 기능 | 유지 |
| **MCP 도구 (Tool)** | 핑크/빨강 | 외부 연결 (Gmail·택배) | 유지 |
| **규칙 (Rule)** | 노랑 | 제약 | 유지 |
| **에이전트 (Agent)** 🆕 | 보라 | Claude 두뇌 (판단·생성) | **추가** |
| **하네스-중복체크** 🆕 | 회색/주황 | 이미 처리한 것 제외 | **추가** |
| **하네스-검증** 🆕 | 회색/주황 | 결과 맞나 확인, 아니면 중단 | **추가** |
| **하네스-🚨승인게이트** 🆕 | **빨강(강조)** | 실행 멈추고 사용자 승인 받기 | **추가·데모 핵심** |

> 하네스 노드는 "제어(Control)" 카테고리로 묶어도 됨. 승인게이트는 눈에 띄게(🚨).

## 5. 데모 시나리오 (CS 신입) — 프론트가 만들 흐름

**주인공**: CS 신입 김주임. "클로드로 CS 업무 효율 높여봐" 지시받음.

**화면 순서 (영상 흐름)**:
1. **자연어 입력 화면** — 김주임: *"CS 담당인데, 컴플레인 오면 긴급도 보고, 배송문제면 조회해서 사과+쿠폰 보내는데, 고객한테 보내기 전 나한테 먼저 확인받고 싶어"* → Claude가 구성 제안 → "캔버스에서 편집하기"
2. **캔버스 화면** — 조립된 워크플로우 그래프 표시 (트리거→에이전트(긴급도)→중복체크→MCP(택배)→검증→에이전트(사과문)→🚨승인→MCP(Gmail발송)). 김주임이 노드 보고 편집.
3. **키 연결 팝업** — 도구에 키 필요 → "택배는 API 키가 필요해요. [여기서 발급 →] 붙여넣어 주세요" 팝업 → 붙여넣기 (섹션 6)
4. **▶ 실행** — 노드들이 순서대로 실행되는 애니메이션 (대기→실행중→완료)
5. **🚨 승인 팝업** — *"분노 95%, 대전HUB 체류. 사과+쿠폰 발송할까요? [승인] [취소]"* → 김주임 [승인]
6. **완료** — 실제 사과메일 발송됨, 노드 전부 완료 표시
7. **💾 .claude 저장 / 🖼️ 갤러리 공유** — 만든 봇을 저장·공유

## 6. 크리덴셜(키) 연결 UX — "붙여넣기 팝업" (범용)

도구마다 UI를 따로 안 만듦. **카탈로그 메타데이터로 팝업을 자동 생성:**

```
카탈로그 항목 예:
{ "name":"택배조회", "auth":{
    "type":"api_key", "label":"API 키",
    "get_url":"https://info.sweettracker.co.kr", "help":"가입 후 발급" } }
        ↓ 프론트가 이 데이터로 팝업 자동 생성
┌─────────────────────────────────────┐
│ 🔑 택배조회 연결                      │
│ 이 도구는 API 키가 필요해요.          │
│ [여기서 발급받기 →]                   │
│ [_______________________] (붙여넣기)  │
│                    [연결하기]         │
└─────────────────────────────────────┘
```

- `auth.type: "api_key"` → **붙여넣기 팝업** (대부분 도구: 택배·텔레그램·MCP)
- `auth.type: "oauth"` → **"연결하기" 버튼** (Gmail 등 소수)
- 넣은 키는 **로컬에 저장** → 그 도구 쓰는 모든 워크플로우에서 재사용 (매번 안 물어봄)

## 9. 프론트 ↔ 백엔드 API (대략 감만)

정확한 스펙은 백엔드랑 맞추되, 대략 이런 호출이 있음:

**호스팅 백엔드 (①):**
- `POST /assemble` (자연어 → 그래프 JSON)
- `GET /catalog` (MCP·도구 목록 + 키 메타데이터)
- `GET/POST /gallery/...` (목록·발행·가져오기)

**로컬 실행기 (②, localhost):**
- `GET /graph` (.claude 읽어서 그래프)
- `POST /save` (그래프 → .claude 저장)
- `POST /run` (워크플로우 실행 시작)
- `GET /run/{id}/status` (실행 상태 폴링 — 노드별 진행·승인대기 여부)
- `POST /run/{id}/approve` (승인게이트 통과)
- `POST /credential` (키 붙여넣기 저장)

> 실행 진행은 **폴링(status 계속 조회)** 또는 SSE로 받아 노드 상태를 갱신하면 됨.

## 10. 기술 스택 (프론트)

- **React + TypeScript + React Flow** (노드 캔버스) → **Vercel 배포**
- 인증 = **Clerk** (프리빌트 컴포넌트)
- 호스팅 백엔드(FastAPI)랑은 REST + **CORS**(Vercel 도메인 허용)
- 로컬 실행기랑은 **localhost REST** (사용자 PC에서 도는 것과 통신)

---

## 11. 🗄️ 백엔드(갤러리) 파트 — 주니어 담당 [상세]

> 이건 **호스팅 백엔드(FastAPI, EC2)** 얘기. 로컬 실행기(팀장 담당)랑 별개.

**만들 것 (갤러리 = 공유 기능):**
- **인증**: Clerk 토큰 검증 미들웨어 (Clerk가 로그인 다 해줌 → 우리는 "이 토큰 유효?"만 확인)
- **갤러리 CRUD API** (FastAPI): 워크플로우/스킬 **발행**, **목록 조회**, **가져오기**, **삭제**, **태그**, **좋아요**
- **MCP 카탈로그 API**: 지원 도구 목록 + 키 메타데이터 반환 (프론트가 "키 붙여넣기 팝업" 만들 때 씀)
- **DB(RDS Postgres)**: users·workflows·skills·tags·likes 테이블 (SQLAlchemy + Alembic)

**저장 데이터 (여기가 중요 — 개인 데이터는 여기 안 옴):**
| RDS(서버, 갤러리) | 로컬 SQLite(사용자 PC, 팀장 담당) |
|---|---|
| users·workflows·skills·tags·likes (공유) | 처리이력·실행상태(승인 중단/재개) (개인) |
| → 남들과 공유되는 것만 | → 사용자 메일/컴플레인은 저장 안 함 |

**규모감**: 갤러리는 표준 CRUD라 초보도 할 만함. 인증은 Clerk가 해주니 직접 안 짬.

## 12. 🧪 비개발 파트 — 데이터·리서치·영상·QA

- **택배 API**: 스마트택배(스윗트래커) 가입 → API 키 발급 (리드타임 확인). 실제 송장번호로 조회 테스트.
- **CS 샘플 데이터**: 데모용 "화난 고객 컴플레인 메일" 문구 몇 개 작성 (긴급도 높은 것/배송지연 등).
- **프롬프트 설계**: 긴급도 판단·사과문 생성 에이전트의 프롬프트 문구 다듬기 (팀장과 협업).
- **데모 영상**: CS 신입 시나리오 촬영·편집 (3~5분). 승인게이트 팝업 장면이 클라이맥스.
- **QA**: 시나리오대로 한 경로 완주되는지 반복 테스트, 깨지는 지점 리포트.
