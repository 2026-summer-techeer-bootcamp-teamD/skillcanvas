# 📄 PRD — oh my skill *(가제 · 내부명 SkillCanvas)*

> **한 줄 정의:** 내가 커스텀한 AI 스킬을 **부품처럼 쌓아두고**, 드래그로 **워크플로우에 데려와 자동화**하고, **갤러리로 공유·재사용**하는 플랫폼.
>
> **누구를 위해:** Claude로 업무 자동화하고 싶은 **비개발자·초보자**, 스킬을 여러 개 관리하는 사람.
>
> **차별점 한 줄:** n8n은 '데이터 흐름'을 그리지만, 우리는 **당신의 Claude 스킬 자체를 그리고, 로컬 파일과 양방향으로 동기화**한다.

*(문서 버전: v3 상세 · 작성일 2026-07-03 · 대상 독자: 팀원 전원, 특히 개발 초보자)*

## 📖 0. 용어사전 (여기부터 읽으세요)

> 초보 팀원용. 모르는 단어 나오면 여기로.

- **Claude Code** — 터미널에서 도는 AI 개발 도구(CLI). 우리가 "실행 엔진"으로 재사용.
- **스킬(Skill)** — Claude에게 시키는 "나만의 기능". `.claude/skills/이름/SKILL.md` 파일 하나 = 스킬 하나.
- **MCP** — Claude가 외부 도구(노션·구글시트·깃허브)에 손을 뻗게 해주는 **연결 어댑터**. `.mcp.json`에 등록.
- **룰/권한** — Claude가 뭘 해도 되고 안 되는지 정하는 설정. `.claude/settings.json`.
- **`.claude` 폴더** — 위 스킬·MCP·룰 파일들이 모여 있는 내 컴퓨터 안의 폴더.
- **워크플로우(Workflow)** — 스킬·도구들을 **순서대로 엮은 자동화 흐름**. (예: 회의록 받기 → 요약 → 노션에 쓰기)
- **노드(Node)** — 화면에 뜨는 박스 하나. 스킬/MCP/룰/워크플로우 단계를 나타냄.
- **그래프(Graph)** — 노드들과 그 연결(선)을 합친 것. = 화면에 그려지는 그림.
- **파싱(Parsing)** — **파일 → 그림**으로 읽어들이는 것. (파일 내용을 이해해서 노드로 바꿈)
- **직렬화(Serialize)** — **그림 → 파일**로 다시 써넣는 것. (노드 편집을 파일에 저장)
- **양방향 동기화** — 파싱 + 직렬화. 파일 고치면 그림 바뀌고, 그림 고치면 파일 바뀜.
- **DAG** — 화살표가 한 방향으로만 흐르는 순서도(순환 없음). 워크플로우의 실행 순서.
- **CLI** — 터미널에 명령어 쳐서 프로그램 부르는 방식. (예: `claude -p "..."`)
- **로컬 실행기(심부름꾼)** — 내 컴퓨터에서 도는 작은 프로그램. 브라우저 대신 `claude` CLI를 쳐주고 파일을 읽고 씀.
- **BaaS** — 백엔드 기능(인증 등)을 대신 해주는 서비스. 우리는 인증을 **Clerk**(BaaS)에 맡김.

## 1. 문제 의식 (왜 만드나)

- 요즘 다들 Claude로 업무 자동화(스킬)를 만들려고 함.
- 스킬을 만들려면 `.claude` 폴더에 **여러 파일**(SKILL.md·MCP·룰)이 쌓임 → **눈에 안 보이고, 뭘 붙여야 할지 모름.**
- 비개발자는 진입장벽 때문에 **시작도 못 함.**
- 게다가 대화형 Claude는 **매번 다시 설명해야 하고, 끝나면 사라짐** → 재사용·자동화·공유가 안 됨.

> **핵심 고통:** "내 AI가 뭘로 구성됐는지 안 보이고, 한 번 만든 걸 재사용·자동화·공유할 수가 없다."

## 2. 핵심 컨셉 — 만들고 · 조립하고 · 공유

```
   📦 부품 창고            ⚙️ 워크플로우           🖼️ 갤러리
   나만의 스킬 커스텀   →   부품 엮어 자동화    →   올리고 / 가져오기
   (.claude 동기화)        (자동생성·실행)        (커뮤니티 공유)
        ▲                                              │
        └──────────── 가져온 워크플로우를 내 것으로 저장 ─┘
```

- 창고에서 **내 스킬을 커스텀** → 워크플로우로 **드래그해 조립·실행** → 갤러리에 **올리거나 남 것을 가져옴**.
- 결과는 **파일로 저장돼 재사용** → *대화는 사라지지만, 내 부품은 자산이 된다.*

## 3. 핵심 기능 명세

> 각 기능: **뭐하나 / 사용자 흐름 / 우리가 만드는 부분 / 우선순위(P0~P2)**

### 📦 ① 부품 창고 — 나만의 스킬 관리

**뭐하나**: 내 `.claude`(스킬·MCP·룰)를 노드로 보여주고, 커스텀하고, 파일과 양방향 동기화.

**사용자 흐름**
1. 로컬 실행기 켜면 → 내 `.claude` 파일을 읽어 **노드로 시각화** (파일→그림)
2. 노드 클릭 → **파일 직접 편집** or **자연어로 "이렇게 고쳐줘"** (AI가 수정)
3. **💾 저장** → 실제 `.claude` 파일에 반영 (그림→파일)
4. 자연어로 요청 → 필요한 **MCP 자동 추천·등록** (큐레이션된 카탈로그에서)

**우리가 만드는 부분**
- 파서(파일→그래프), 직렬화기(그래프→파일, **본문 보존**) ⭐우리 핵심
- MCP 카탈로그(레지스트리)
- (자연어 이해·생성은 Claude, 검증·형식맞춤은 우리)

**우선순위**: `P0` (시각화·양방향동기화·MCP추천)

---

### ⚙️ ② 워크플로우 — 부품 조립 + 실행

**뭐하나**: 스킬·도구를 순서대로 엮어 자동화 흐름을 만들고 실행.

**사용자 흐름**
1. 자연어 입력 → **워크플로우 그래프 자동 생성** (AI 추천)
2. **창고 부품을 드래그로 불러오기** · 노드 드래그로 자유 배치
3. 노드 직접 추가 → 자연어를 skill/MCP로 **자동 매핑**
4. 노드 클릭 → 자연어로 수정 (AI 분석·적용)
5. **▶ 실행(테스트)** — 단계별로 진짜 동작 (AI 단계는 실제 Claude)
6. **⏰ 스케줄 걸기** — "매일 9시 실행" (로컬 실행기 켜져 있을 때)
7. **💾 배포용 config 내보내기** — workflow spec + agent config + deployment config

**우리가 만드는 부분**
- 워크플로우 그래프 모델링 + **DAG 실행 오케스트레이션**(순서·상태) ⭐우리 핵심
- 자연어→노드 매핑의 **제약·검증·에러복구**
- config 변환기, 로컬 스케줄러(APScheduler)

**우선순위**: `P0`(생성·편집·실행·config) / `P1`(스케줄)

---

### 🖼️ ③ 갤러리 — 공유·재사용

**뭐하나**: 완성한 워크플로우/스킬을 올리고, 남의 것을 내 GUI로 가져옴.

**사용자 흐름**
1. 로그인(Clerk) → 내 워크플로우/스킬을 **갤러리에 발행** (태그 분류)
2. 남의 워크플로우를 **내 GUI로 가져오기** = 그래프 JSON을 캔버스에 로드
3. **"내 것으로 저장"** → 그 안의 스킬·MCP가 내 로컬 `.claude`에 반영 (기존 동기화 재사용)

**우리가 만드는 부분**
- 갤러리 CRUD(발행·목록·가져오기·삭제·태그·좋아요) + DB 설계
- 인증은 **Clerk에 위임**(토큰 검증만 우리)

**우선순위**: `P1` (확장)

## 4. 아키텍처 (하이브리드: 화면·공유는 웹, 파일·실행은 로컬)

> **핵심 원리:** 로컬 파일을 만지는 도구는 순수 웹사이트가 될 수 없다(브라우저 보안).
> → **웹(화면·갤러리) + 다운로드하는 로컬 실행기(파일·CLI 심부름) + Claude(두뇌·실행)** 로 나눈다.

```
        [GitHub] → [GitHub Actions] ──(백엔드 이미지 빌드·배포)──┐
                                                               ▼
   ┌─ Vercel ──────┐              ┌─ EC2 (Docker) ─────────────────┐
🧑─▶│ React+ReactFlow│──(CORS/REST)▶│ Nginx → Uvicorn → FastAPI        │──▶ Clerk(인증)
   │  노드 캔버스    │  갤러리·추천  │   · 갤러리 CRUD · 카탈로그        │──▶ RDS(PostgreSQL)
   └───────┬───────┘              │   · 토큰검증 · AI추천             │
           │                      └──────────────────────────────────┘
           │ localhost (파일·실행)
   ┌───────▼──────────────────────────────────────┐
   │ 🖥️ 로컬 실행기 (사용자 PC, 다운로드형)          │
   │  FastAPI + APScheduler                        │
   │   · 파서/직렬화기 (.claude ↔ 그래프)           │
   │   · 워크플로우 엔진 (DAG 실행)                  │
   │   · Claude 브리지: execFile('claude', ...)     │
   │   · config 내보내기 · 스케줄러                 │
   └──────────┬───────────────────────┬────────────┘
              ▼                        ▼
       🧠 Claude (API/CLI)     🔌 MCP: Notion·Sheets·GitHub
```

**왜 이 구조인가**
- **실행 엔진 안 만듦** — 로컬 실행기는 `claude` CLI를 대신 쳐주는 **심부름꾼**일 뿐, 실제 실행은 Claude.
- **웹이 로컬 파일 직접 못 만짐** → 다운로드형 로컬 실행기 필요 (Docker Desktop·Ollama와 같은 방식).
- **공유는 가벼운 JSON만** → 백엔드는 저장·태그·인증검증만.
- **가져오기 = GUI에 로드** → AI 추천 렌더 코드 재사용.

**브라우저는 두 곳과 통신**
- **EC2 백엔드** ← 갤러리·카탈로그·AI추천 (인터넷)
- **로컬 실행기(localhost)** ← 파일 읽기/쓰기·워크플로우 실행 (내 컴퓨터)

**배포 방식** — 웹사이트의 [로컬 실행기 다운로드] 버튼으로:
- 설치파일(.exe/.dmg) 더블클릭(비개발자) · `npx` 명령어(개발자)
- 한 번 설치하면 그 뒤론 웹에서 원클릭으로 config 생성·실행

> 웹↔로컬 연결 상세(CLI 호출·심부름꾼·다운로드)는 `발표_예상질문_대비.md` Q1 참고.

## 5. 기술 스택 (확정)

| 층 | 기술 | 왜 |
|----|------|-----|
| **프론트** | React + TypeScript + **React Flow** → **Vercel** | 노드 캔버스 표준, CDN·배포 자동, 백엔드와 분리 |
| **인증** | **Clerk** (BaaS) | 인증은 기본기+보안리스크 → 위임, 프리빌트 컴포넌트 |
| **백엔드** | **FastAPI + Uvicorn** (Python) | 가볍고 async·subprocess 궁합, 카탈로그+엔진 통일 |
| **DB** | **AWS RDS (PostgreSQL)** | 그래프 JSON을 `jsonb`로 저장·쿼리, 스키마 자체 설계 |
| **ORM/마이그레이션** | SQLAlchemy + Alembic | DB 층 직접 소유 (백엔드 어필) |
| **배포** | **EC2 + Docker + Nginx** | 직접 DevOps, Nginx 리버스프록시·HTTPS |
| **CI/CD** | GitHub + GitHub Actions | 빌드·배포 자동화 |
| **로컬 실행기** | FastAPI + APScheduler + Claude CLI | 파일·실행·스케줄 |
| **AI 두뇌** | Claude (API 또는 로컬 CLI) | 자연어 이해·생성 |
| **관측(선택)** | Sentry | 에러 추적 |

**중복 방지 메모**
- Nginx vs ALB → 단일 EC2 + Nginx(문지기) 구성이라 ALB 불필요. (트래픽 커지면 로드맵)
- Redis·Celery → 코어엔 없음. **무인 트리거 확장 시** 도입 (로드맵).

## 6. 기술 상세 — 각 기술 "누가 하나" (심사 대비 핵심)

> 초보 팀원이 "이거 Claude가 다 하는 거 아냐?" 헷갈릴 때 이 표.

| 기술 | 우리 | Claude | 라이브러리 |
|------|:---:|:---:|:---:|
| **양방향 포맷 보존 동기화**(파서+직렬화기) | ●●● | | |
| **워크플로우 모델링 + DAG 실행** | ●●● | | |
| **MCP 카탈로그(레지스트리)** | ●●● | | |
| **자연어 → 구조 매핑** | ●●(제약·검증·에러복구) | ●●(이해·생성) | |
| **부품↔워크플로우↔갤러리 루프** | ●●● | | |
| 노드 화면 렌더링 | ● | | ●● React Flow |
| AI 두뇌(자연어 이해) | | ●●● | |

**핵심 방어 논리**
- "Claude로 개발한 것"(개발 도구) ≠ "제품이 실행 중 Claude를 부르는 것"(아키텍처). 둘은 다름.
- 우리 백엔드 무게중심 = **인증 아님** → **① 동기화 엔진 ② 워크플로우 오케스트레이션 ③ DB 층(자체 설계)**.

### config 내보내기 원리
워크플로우 그래프를 세 덩어리로 변환:
- **workflow spec** — 노드·순서(DAG)
- **agent config** — 각 AI 단계가 뭘 하는지
- **deployment config** — 필요한 MCP·권한·트리거
→ 그래프를 읽어 이 셋으로 뽑는 게 우리 코드. Claude Code가 이 config를 읽고 실행.

## 7. 데이터 모델 (ERD 초안)

> 인증은 Clerk가 담당 → 우리 DB는 `clerk_user_id`로 유저를 참조.

```
users
  id PK · clerk_user_id(uniq) · display_name · avatar_url · created_at
      │1
      │
      │N
workflows                          skills
  id PK                              id PK
  owner_id FK→users                  owner_id FK→users
  title · description                name · description
  graph_json (jsonb) ⭐              skill_md (text)
  is_public · created_at             frontmatter_json (jsonb)
      │N          │N                 created_at
      │           │
   workflow_tags  likes / imports
      │N          (user_id FK, workflow_id FK, created_at)
      │
   tags(id PK, name)   ←  skill_tags(N:M)

mcp_catalog  (큐레이션 레지스트리)
  id PK · key · name · npx_command · auth_type · env_vars_json · description
```

**핵심 포인트**
- `graph_json`·`frontmatter_json` → Postgres **jsonb**로 저장(그래프·설정을 통째로, 쿼리 가능).
- `mcp_catalog` → AI가 추천할 수 있는 MCP는 여기 등록된 것만 (제품 자산).
- M:N 태그, 좋아요/가져가기 카운트 → 갤러리 도메인.

## 8. 화면 설계

**모드 전환**: 📦 부품 창고 ↔ ⚙️ 워크플로우 ↔ 🖼️ 갤러리

```
[부품 창고]                          [워크플로우]
┌──────────────┬──────────┐        ┌──────────────┬──────────┐
│  노드 캔버스   │ 편집 패널  │        │ 오팔풍 캔버스  │ 노드 편집  │
│ 스킬─MCP─룰   │ ·파일편집  │        │ 트리거→AI→도구│ 자연어수정 │
│ (구성 그래프) │ ·자연어수정│        │ 드래그·✕삭제  │          │
│ [자연어 추천] │ ·💾저장    │        │ [▶실행][💾config]        │
└──────────────┴──────────┘        └──────────────┴──────────┘

[갤러리]
┌───────────────────────────────┐
│ 카드 목록(태그) · [가져오기]     │
│ → 그래프 JSON을 내 캔버스에 로드 │
└───────────────────────────────┘
```

- 노드 = 실제 파일과 1:1 (스킬/MCP/룰)
- 선 = "이 스킬이 이 MCP·룰을 쓴다" → **흐름이 아니라 구성**
- 워크플로우 노드 = 실행 단계 (트리거/ai/도구/출력)

</aside>

## 9. 이번 2주 범위 (스코프) + 성공 기준

**✅ 만든다 (In)**
- 부품 창고: 파일↔그래프 양방향 동기화(본문 보존), 자연어 MCP 추천·등록, 노드 편집
- 워크플로우: 자연어 자동생성, 드래그 편집, 노드 자연어 수정, ▶실행, config 내보내기
- 로컬 스케줄 트리거 (실행기 켜졌을 때)
- 갤러리: 로그인(Clerk), 발행·가져오기·태그
- 예시 스킬 **1개는 Claude Code에서 진짜 실행**까지 시연

**❌ 안 만든다 (Out — 로드맵으로 발표만)**
- 무인 트리거(꺼져 있어도 실행) · Celery/Redis · secrets vault
- 실시간 파일 감시 · 실행 관측(트레이스·비용) · 자가 개선 루프

**성공 기준 (데모에서 보이면 성공)**
1. 자연어로 말하면 → 스킬/워크플로우 노드가 자동 생성
2. 노드를 고치면 → 실제 `.claude` 파일이 바뀜 (양방향)
3. 터미널에서 만든 스킬이 → GUI에 노드로 뜸
4. ▶실행 → AI 단계가 실제 Claude로 동작
5. 갤러리에서 가져오기 → 내 캔버스에 로드

## 11. 개발 순서 / 마일스톤 (2주)

**1주차 — 코어**
- Day 1–2: 로컬 실행기 뼈대(FastAPI) + `.claude` 파서/직렬화 (양방향 동기화)
- Day 3–4: 프론트 노드 캔버스(React Flow) + 부품 창고 연동
- Day 5: 자연어→MCP 추천(Claude 브리지) + 카탈로그

**2주차 — 워크플로우 + 갤러리 + 마감**
- Day 6–7: 워크플로우 생성·편집·실행(DAG) + config 내보내기
- Day 8: 로컬 스케줄 트리거 + 백엔드(EC2·Docker·Nginx·RDS) 배포
- Day 9: 갤러리(Clerk 로그인·발행·가져오기·태그)

> 원칙: **코어 먼저 → 갤러리는 그 위 레이어.** 시나리오 1개만 진짜 실행, 나머지는 시연용.
