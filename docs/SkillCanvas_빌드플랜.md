## ⚠️ 먼저 읽어야 할 "가정 & 미결정" (확정사실과 구분)

아래는 **내(PM) 추측/제안**이며 팀장 확정이 필요함. 확정 전까지 플랜의 해당 부분은 잠정.

1. **[결정됨 ✅ 2026-07-04] 로컬 실행기 = Python/FastAPI 로 확정.** PoC(JavaScript/Node)는 "참고 레퍼런스"로만 두고, 로컬 실행기는 Python으로 새로 구현.
   → 이유: 팀이 파이썬 고정 + 어차피 제품은 새로 빌드(프론트 React·백엔드 FastAPI) + PoC는 production 코드가 아니라 프로토타입 → 언어 통일(전부 Python)이 팀 유지보수·일관성에 유리. 로컬 로직(파서·직렬화·CLI호출·NL→그래프)은 몇백 줄 규모라 PoC를 정답지 삼아 Python 재구현 비용 작음 (`subprocess`·`sqlite3`·FastAPI·APScheduler로 전부 가능).
2. **[추측] 택배조회 API는 스윗트래커(구 스마트택배) 또는 각 택배사 API 중 발급 빠른 것.** 실제 발급 소요·무료 한도 미확인 → N2가 D+2까지 리서치.
3. **[가정] 데모 Gmail은 팀이 만든 테스트 계정 2개(고객役/CS役).** 실제 고객 데이터 아님 → 프라이버시·안전.
4. **[가정] 발표 영상 분량 3~5분, CS 워크플로우 1회 완주 + 조립 장면 일부.**

## 2. 의존성 순서 (뭐 먼저 → 뭐)

```
[코어 크리티컬 패스 — 팀장]
A2(SQLite스키마) ─┬─▶ A3(중단/재개 엔진) ─▶ B3(승인게이트) ─┐
                  │                                          │
A4(에이전트노드) ─┤                                          ├─▶ C3(CS 배선) ─▶ G4(영상)
                  │                                          │
B1(중복)·B2(검증)─┘   C1(Gmail)·C2(택배) ─────────────────────┘

[병렬 가능 — 코어와 독립]
J: E1(Clerk) ─▶ E2(갤러리CRUD) ─▶ E3(태그) ,  D1·D3(캔버스·진행)
N1: G1(데이터) ─▶ G2(프롬프트)  |  N2: G3(택배리서치) ─▶ G4(영상)
T: F2(RDS) ─▶ F1(EC2배포)  (2주차 후반)
```

**한 줄 원칙: `A2 → A3 → 하네스(B) → CS연동(C) → 영상(G4)`가 최우선 직렬 체인.** 이게 밀리면 데모가 안 나옴. 갤러리·DevOps·데이터는 이와 병렬로 굴림.

**주의:** B3(승인게이트)은 A3(중단/재개) 위에 얹히므로 **A3를 절대 먼저**. C3(CS 배선)은 A·B·C1·C2가 다 있어야 완주.

| R2 | **Gmail OAuth** — 발급·스코프·발송권한 | 🔴높음 | 주말(7/12~13) 통째 배정. 안 풀리면 **폴백: Gmail API 앱비밀번호/서비스계정 or 읽기는 실제·발송은 "발송됨" 시뮬**(영상이라 허용). 테스트 계정으로 프라이버시 회피 |
| R3 | **택배 API 발급·선정** — 리드타임·승인대기 | 🟠중 | **7/4 최우선 착수**(리드타임 김). 후보 2개 동시 발급. 최악 시 **고정 송장 1건 응답을 모킹**(영상용). N2 전담 |
| R4 | **EC2+Docker+Nginx 배포** | 🟠중 | 2주차 후반 배정. 안 되면 **폴백: 백엔드도 로컬/Railway 등 간이 호스팅**으로 영상 촬영, 발표선 "EC2 구성도"로 설명. DevOps는 데모 필수경로 아님 |
| R5 | **실데이터 한 경로 완주** — 여러 연동이 한 번에 안 맞음 | 🔴높음 | 7/17까지 러프 완주, 7/18~21 안정화. **모든 값 하드코딩 가능(고정 고객·고정 송장·고정 프롬프트)**. 영상은 "완벽한 한 번"만 필요 |
| R6 | **인력: 실질 개발 1명(팀장)에 코어 집중** | 🟠중 | 갤러리·프론트·데이터를 J·N에 최대한 오프로드. 팀장은 A·B·C·F에만. PoC 재사용으로 절대량 축소 |
| R7 | **PoC(JS/Node) 로직을 Python으로 이식** | 🟡낮 | 로컬 실행기 Python 확정(2026-07-04). PoC를 정답지 삼아 이식 → 규모 작음(몇백 줄), Claude 포팅 보조 가능 |

## 5. 팀별 트랙 (병렬)

| 트랙 | 주 담당 | 뭘 하나 (병렬) |
|------|--------|----------------|
| **① 코어 엔진·하네스·CS·DevOps** | 팀장 T | A(엔진)·B(하네스)·C(Gmail/택배)·F(배포). **크리티컬 패스 전담.** |
| **② 갤러리·인증·프론트** | 주니어 J | E(Clerk·CRUD·태그) + D(캔버스·팝업UI·진행표시) + F3(Vercel). **코어와 독립**이라 안 막힘. |
| **③ 데이터·프롬프트** | 비개발 N1 | G1(테스트계정·샘플메일)·G2(프롬프트 3종)·G5(QA). 팀장 연동 지점만 맞추면 됨. |
| **④ 택배리서치·데모영상** | 비개발 N2 | G3(택배API 발급)·G4(대본·녹화·편집)·G5(QA). **G3는 D+0 최우선.** |

> **초보 배려:** J·N은 "코어 완성 대기" 없이 첫날부터 각자 것 착수. 진입장벽 낮은 몫(Clerk 프리빌트, 데이터 준비, 영상)으로 배정. 애매하면 각 태스크 DoD를 기준으로 "이게 되면 끝".

## 6. 스코프 가드 (In vs Out)

**✅ In — 2주 코어 (반드시)**
- 에이전트 노드 (긴급도·사과문 2종)
- 하네스 3종: 중복체크·검증·**승인게이트(중단/재개)**
- 로컬 SQLite (처리이력·실행상태)
- CS 연동: Gmail 읽기/발송 + 택배조회
- CS 워크플로우 **한 경로 완주** (①~⑩)
- 갤러리(Clerk 로그인·발행·가져오기·태그) + PoC 재사용분(파서/직렬화·NL→그래프·MCP추천)
- 배포(EC2·Docker·Nginx·RDS) — 되면 실물, 안 되면 폴백

**❌ Out — 로드맵 (안 만듦, 발표선 말로만)**
- 조건분기·서브에이전트·병렬 실행·재시도 정책
- 무인 트리거(꺼져 있어도 실행)·Celery/Redis·secrets vault
- 실시간 파일 감시·실행 관측(트레이스·비용)·자가 개선 루프
- 카톡·젠데스크 연동
- 모든 케이스 robust (데모는 한 경로만)

> **새 요구 판정 규칙:** "이게 CS 한 경로 완주 영상에 필요한가?" **No면 무조건 로드맵.** 승인게이트·Gmail·택배·완주 외의 확장은 전부 차단.

### 팀장이 결정해야 할 것 (플랜 확정 전 필요)
1. ~~[스택] 로컬 실행기 = Node vs Python?~~ → **결정됨 ✅: Python/FastAPI** (2026-07-04). PoC(JS)는 참고용, 전부 Python 통일. 이식 비용 작음.
